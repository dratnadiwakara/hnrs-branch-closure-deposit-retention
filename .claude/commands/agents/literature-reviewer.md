You are running the **literature-reviewer** agent. Your inputs are:
- `related-papers/topic-papers.md` — the project-local index of relevant slugs (produced by `/agents/literature-downloader`).
- `C:/OneDrive/github/paper-repo/mds/` — the shared markdown files for those slugs.
- `related-papers/references.bib` — the BibTeX file produced by the downloader (used as-is if present).

You do **not** download PDFs, run any conversion, or access `related-papers/mds/`.

## Phase 0: Verify inputs

1. Check that `related-papers/topic-papers.md` exists. If missing, stop and tell the user:
   > "`related-papers/topic-papers.md` not found. Please run `/agents/literature-downloader` first."

2. Read `related-papers/topic-papers.md` and extract the list of slugs from the **Selected Papers** table.

3. For each slug, verify that a matching `*.md` exists in `C:/OneDrive/github/paper-repo/mds/`. Collect:
   - **Present**: slugs with a markdown file.
   - **Missing**: slugs with no markdown file — warn the user but continue with the present ones.

4. If no slugs are present, stop and tell the user:
   > "No markdown files found in the shared repo for the slugs in `topic-papers.md`. Please re-run `/agents/literature-downloader` to convert the PDFs."

5. Define constants:
   - `REPO_MD    = C:/OneDrive/github/paper-repo/mds`
   - `REPO_CSV   = C:/OneDrive/github/paper-repo/papers.csv`
   - `TOPIC_LIST = related-papers/topic-papers.md`
   - `BIB_PATH   = related-papers/references.bib`
   - `REVIEW     = related-papers/literature-review.md`
   - `CRITIQUE   = related-papers/literature-review-critique.md`

## Phase 1: BibTeX

1. Check whether `BIB_PATH` already exists (the downloader writes it after each phase).

2. **If it exists**: use it as-is for all subsequent phases. Tell the user:
   > "`related-papers/references.bib` was produced by the downloader and will be used directly. Review it before submission."

3. **If it is missing** (downloader was not run, or bib was deleted): build it from scratch.
   - Load `REPO_CSV` and filter to rows whose `slug` appears in the selected slugs from `TOPIC_LIST`.
   - For each matching row, construct a BibTeX entry:
     - Use `slug` as the BibTeX key.
     - Map `type` → BibTeX entry type: `article` → `@article`, `book` → `@book`, `incollection` → `@incollection`, anything else → `@misc`.
     - Include only non-blank fields. Do not guess missing values.
     - Valid BibTeX syntax: comma-separated fields, values in `{}`, no trailing comma after the last field.
   - For any slug whose CSV row has critical fields missing (title, authors, year), fall back to reading the first ~100 lines of its markdown in `REPO_MD` to infer those fields.
   - Write `BIB_PATH` with this header comment:
     ```bibtex
     % Auto-generated by /agents/literature-reviewer from papers.csv
     % Review and correct entries manually before submission.
     ```

## Phase 2: Summarize papers

For each slug in the present list, read `REPO_MD/<slug>.md`:

- Read enough to understand: research question, data and sample, empirical/theoretical approach, main findings, notable limitations.
- **Long markdowns** (equivalent of >~15 pages): launch parallel general-purpose subagents, one per ~5-page chunk of text, each producing a short chunk summary. When all return, synthesize into a single 2–5 sentence paper summary.
- **Short markdowns**: read the abstract, introduction, and conclusion directly.
- Draft a 2–5 sentence summary that covers:
  - What the study asks and why it matters.
  - How it answers the question (design, model, or data).
  - Key results and any notable limitations.
- Link each summary to the paper's BibTeX key (the slug).

Retain all summaries in memory for use in Phase 3.

---

## Phase 3: Draft the literature review

1. **Identify themes**: group the papers into 2–5 thematic clusters based on topic, method, or mechanism. Assign each paper to at least one theme.

2. **Write `REVIEW`** (create or overwrite) using this structure:

```markdown
# Literature Review: [Topic from topic-papers.md]

**Date**: [Today's date]
**Number of Papers Reviewed**: [N]

## Overview

[1–3 paragraphs summarizing the overall state of the literature, major themes,
and why this body of work matters.]

## Thematic Discussion

### Theme 1: [Descriptive name]

[Short paragraph introducing the theme.]

- **[BibKey] Author et al. (Year)**: [Summary — question, method, findings, limitations.]
- **[BibKey] Author & Author (Year)**: [Summary...]

### Theme 2: [Descriptive name]

[...]

## Gaps and Directions for Future Work

[What is missing, tensions across findings, and promising directions,
based only on the reviewed papers.]

## References

[Formatted reference list drawn only from `references.bib`. For each entry:
Authors. (Year). Title. Journal/Source. BibTeX key: [key].]
```

- The References section must cite only keys present in `BIB_PATH`.
- Write as a continuous narrative, not just bullet lists.

---

## Phase 4: Critique

1. Launch a **general-purpose subagent** with the following role:

   > You are a **senior researcher** evaluating the quality of a literature review.
   > Read `related-papers/literature-review.md` and `related-papers/references.bib`.
   > Assess:
   > - Coverage: are major strands well represented given the papers available?
   > - Thematic organization and coherence.
   > - Balance and fairness in summarizing results.
   > - Clarity and readability.
   > - Quality and correctness of the references section relative to the .bib file.
   >
   > Provide:
   > - A short overall assessment (2–4 sentences).
   > - 3–8 major comments (structure, themes, missing connections, misinterpretations).
   > - 5–15 minor comments (clarity, style, ordering, small corrections).
   > - Concrete, actionable suggestions for improving the review.

2. Save the critic's output as `CRITIQUE` with these headings:
   - `## Overall Assessment`
   - `## Major Comments`
   - `## Minor Comments`
   - `## Suggested Revisions`
